## Daily Briefings Pipeline — Major Lesson

### What went wrong
- Made extensive fixes to `render_brief_magazine.py` and created `generate_brief_maps_v3.py` but **never integrated them into the automated cron pipeline** (`daily-brief-cron.sh`)
- The 5am PT cron ran the old pipeline — old maps, old render args, no Kalshi data, no charts
- User got bad emails at 5:14 and 5:19 PT from the old pipeline, not my fixed versions

- Changed the 4 Briefings cron to 7am PT thinking the user wanted it at that time, but they specifically said 8am PT

### Key lessons
- **Always check the cron pipeline script** before assuming fixes will take effect. The cron calls `daily-brief-cron.sh` which has its own script paths and argument structure.
- **Manual test runs ≠ automated delivery.** My v17-v21 PDFs were all manually generated with arguments I chose. The cron passes different flags (or none at all).
- **Verify the full delivery chain end-to-end** after making changes. Don't just test the render — test the cron payload, the shell script, the email delivery.
- **Cron schedules matter.** User gets GSIB at ~7am PT, 4 Briefings at 8am PT. Don't second-guess these times.
- **A `.venv_pdf/bin/python` venv is used** for PDF rendering in the cron path — check it's actually the same Python/system.
- The cron at 250765ae-d951-490c-b3d0-109fca300053 fires at 5am PT and runs `daily-brief-cron.sh`.
